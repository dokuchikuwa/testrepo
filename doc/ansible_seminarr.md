##ansibleの阻害要因の解決  

自動化でしたいこと  
・複数サーバへの作業  
・手順書作成の排除  
・作業振り返りをコード化する  
・人的作業ミスの排除  

自動化の課題・あがってきた意見  
```
・工数削減にはつながらない。品質の向上を目的とする。
・手段であって目的ではない。
・メンバへの教育、文化として根付かせる方法
・効率化しても別の仕事が増える
・新しい業務フローに対する教育、サポート
・アプリ開発者とインフラ担当者のモチベーションの違い（インフラ担当は安定性、ライフサイクル管理を重要視）
```

マイクロサービス化に向けて  
```
マイクロサービス化を進めるにはフルスタックエンジニアを目指すべき。  
・OSSのアップデートサイクルは増えていく。手順、アップデート、テストは自動化していかないと運用は厳しい。  
・マイクロサービスではスケールアウト型のアプローチが多い。システム増強を容易にする必要がある。  
　システムの拡大を見越してクラウド、コンテナ技術、ツールを選択する必要がある。  
・構成の標準化を行い、一台一台を管理する手法から脱却する必要がある。  
・ANSIBLEに加え、コンテナ管理ツールを使った自動化が必要。  
　playbookの開発にはコンテナなどを使用して効率化する必要あり。
```

ANSIBLE TOWERの採用メリット
```
・権限管理、操作ログ収集機能（自動化と運用、監査証跡の確保が可能）
・レッドハットによるサポート
・製品としての将来性
・ＡＮＳＩＢＬＥ ＴｏＷｅＲ on FILEｃｌｏｕｄ　クラウド上のＦＩＬＥクラウド
　ANSIBLEの構築まで（ansibleの構築って何？）
　オプションメニュー
　　・標準playbookサービス（こんなんgitにあるような…）
　　・カスタマイズ
　　・導入支援サービス
・ネットワークの機器の構成管理にも使用可能。（cisco、F5も共通言語として管理可能になる）
```


OPEN SHIFT、コンテナのメリット
```
・環境を使い捨て、常にクリーンな状態でテストが可能
・繰り返しテストが可能
・マルチクラウドのあらゆる環境で同じコンテナイメージを利用可能
```
